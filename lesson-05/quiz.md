# L5 Quiz:记忆与状态

**Q1** capstone 里那份累积的 `messages` 属于哪种记忆?它为什么活不过一次程序重启?

**Q2** 你 capstone 里的 `REFUNDED` 集合,是"记忆"还是"状态"?说清判断依据。

**Q3(核心)** "LLM 是无状态的,但 agent 能记住跨会话的用户偏好。"这两句矛盾吗?
用"存 + 取"的读写闭环解释,长期记忆到底是怎么实现"记得"的。

**Q4** 要给 agent 加长期记忆,"取"这一步通常把读出来的记忆放进 messages 的哪个位置?为什么放那?

**Q5(设计+连回旧课)** 你决定用一个 memory.json 存长期记忆。随着用户聊了几百轮,你不可能把
所有历史都存下来、也不可能每次全塞回 prompt。这引出的两个问题,分别对应后面哪两节课?

---

## 参考答案(做完再看)

**A1** 短期记忆 / 工作记忆。因为 `messages` 是内存里的 Python 列表,进程结束就随内存释放,
没有落到任何持久层(文件/DB)。

**A2** 状态(state)。依据:它的作用是给【代码逻辑】用的——`if order_id in REFUNDED` 来防重复退款,
这份数据根本没拼进 messages 给模型看。记忆是给模型看的,状态是给代码用的。

**A3** 不矛盾。模型每次调用确实无状态、不记得上一次。"记得"是 agent(你的代码)造出来的假象:
- 存:上次会话把用户偏好写进 memory.json。
- 取:这次会话启动时读出来,拼进 system prompt 再发给模型。
模型只是"这次的输入里就带着这条偏好",于是表现得像记得。本质是每次把该记的重新喂回它眼前。

**A4** 通常放进 system prompt(或会话开头的一条上下文消息)。因为长期记忆是"背景设定",应在用户
开口前就生效、且对整段对话持续有效;放 system 里最自然,也不占用户每轮的输入。

**A5** ①"存不下这么多/prompt 塞不下" → 对话压缩与上下文管理 = **L6**;
②"该取哪几条最相关" → 按相关性检索记忆 = **L7 RAG**。
