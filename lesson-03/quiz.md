# L3 Quiz:tool use 深入

> 出 quiz 时即落成此文件。含题目、学习者作答、教练反馈。结果:**达标(3 题满分 + 2 题方向对需补全)**。

---

**Q1(概念)** L3 能力大涨(多工具/链式/错误恢复),但 while 循环结构几乎没动。这说明「agent 能力从哪来」?

- **学习者答**:agent 的核心就是循环 ReAct。
- **反馈**:⚠️ 对了一半。ReAct(Reason+Act)这个词用得好,我们的 loop 是它的简化版。但关键结论是:**agent 能力不来自 loop,来自「工具 + 模型」**。loop 是通用固定引擎,想让 agent 会做新事,是加工具、打磨 description,而不是改循环。L2→L3 loop 一行没动,能力全在工具侧增量。

**Q2(链式调用)** 「退掉 B456」两轮调用,第二轮模型怎么知道 B456 已签收可退?

- **学习者答**:先调 query_order,返回的 order 信息 + tool_call_id 组合成 tool message 被 append 进 messages 成为上下文回传给 LLM,LLM 据此了解签收信息。
- **反馈**:✅ 完美。链路每一环都准,连 tool_call_id 的作用都带到了。这是「messages 是记忆」在多轮里的实际运转。

**Q3(错误也是信息)** try/except 转 error JSON 回传 vs 直接 return "退款失败" 给用户,本质区别?

- **学习者答**:错误直接返回用户会一头雾水;作为工具结果回传 LLM,LLM 知道出问题,能解释原因、帮用户解决。
- **反馈**:✅ 对。A123 输出即活证据:模型不但说不能退,还自己给出「等签收再退 / 拒收」方案。

**Q4(description 的威力)** description 改成干巴巴「申请退款」会怎样?模型靠什么决定要不要先查?

- **学习者答**:模型可能给未签收/已取消的订单退款;模型先读 description 再判断是否操作。
- **反馈**:✅ 对。核心:模型决定要不要先查、能不能退,**唯一依据就是 description 里写的字**。description 是最便宜的非侵入式行为旋钮(面试「如何提升工具调用可靠性」第一答案)。

**Q5(串联未来)** 30 个工具全塞 TOOLS 会带来什么新问题?

- **学习者答**:LLM 可能下一大长串指令执行半天;工具太多不好判断用哪个,导致指令不准确。
- **反馈**:⚠️ 答对了准确率一半(工具多易选错)。漏了埋的另一半:**30 个工具的 schema 每轮都随请求发出,本身吃 token → 上下文被工具定义撑大 → 成本/延迟上涨**,是「messages 越滚越长」的孪生兄弟。两问催生两类解法:**工具编排/分组(L4)** 与 **按需检索工具(思路同 L7 RAG,检索对象从知识换成工具)**。

## 总评
达标 ✅。3 题满分、2 题方向对需补全,补的是「能力来自工具」「工具也占上下文」两个承上启下的认知。
