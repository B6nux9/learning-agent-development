## 这节课学到了什么(一句话)
在 L2 单工具 loop 的基础上,发现**agent 的能力增量全在工具侧**——loop 一行没改,靠加工具、打磨 description、处理错误,agent 就从「只会查」升级到「会退款、会链式多步、会从错误里恢复」。

## 核心内容

### 1. 关键认知:能力来自工具,不来自 loop
L2→L3 那个 `while` 循环结构几乎没动,能力却大涨。结论:**loop 是通用固定引擎;想让 agent 会做新事,是加工具 + 写好 description,而不是改循环。** 这条贯穿后面所有课——L4 编排、L8 客服原型都不会再动 loop,只往工具和 prompt 上叠。
(我们这个 loop 是 **ReAct**(Reason+Act,推理与行动交替)的简化版。)

### 2. 多工具:模型靠 description 选工具
工具越多,`description` 质量越生死攸关。写好三要点:说清**做什么**、**何时用**、**何时别用**(把业务规则前置)。模型决定「调不调、先查还是直接退」的**唯一依据就是 description 里的文字**——这是最便宜的非侵入式行为旋钮。

### 3. 错误也是信息(本节最重要的思想)
工具报错 ≠ agent 失败。用 `try/except` 接住异常,把错误文本作为 tool 结果回传给模型,模型会自愈:换参数、改调别的工具、或体面告知用户。
- **坏**:异常击穿 → Python 崩 → 用户看 Traceback。
- **好**:异常 → `{"error": str(e)}` 回传 → 模型解释原因、给方案。

### 4. 链式调用:loop 白送的超能力
「退掉 B456」→ 先 query_order 查状态 → 再 apply_refund。第二步依赖第一步,**需要的新代码为零**:第一轮工具结果 append 进 messages,第二轮模型读到「已签收」自己决定退款。多轮串行编排是**涌现**的,不是额外实现的。
- 区分:**并行**(一轮点多单,互不依赖,如 L2 的 A123+B456)vs **链式/串行**(跨轮,后依赖前)。

### 5. 兜底:循环次数上限
`while True` → `while turn < MAX_TURNS`,超限返回「转人工」话术而非烧 token。生产必备。(学习者在 L2 quiz Q4 自己提出,本节亲手实现。)
