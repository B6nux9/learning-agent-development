# L6 Quiz:上下文管理 + Prompt/结构化输出

**Q1** 为什么多轮对话的 token 消耗是**平方级**而不是线性增长?根源在 LLM API 的哪个特性?

**Q2(反直觉题)** "我换个 128K 超长上下文的模型,就不用管上下文管理了。"
这话错在哪?至少说出两点,其中一点必须涉及**回答质量**。

**Q3** 你提议"每轮让模型顺带输出一段摘要"。除了成本浪费,还有一个**技术实现层面**的硬障碍——
是什么?(提示:想想 API 响应的结构)

**Q4(Prompt)** 下面这句 system prompt 有什么问题?改写它,并说明你用了哪几条原则:
> "你是客服助手。请尽量使用工具来帮助用户,注意不要出错。"

**Q5(结构化输出)** 三个层次(①prompt 请求 ②JSON mode ③schema 强制)各自保证到什么程度?
另外:为什么说 **Function Calling 本质上就是结构化输出**?

**Q6(选型题)** 下面三个场景,分别该用 function calling、结构化输出、还是普通文本?
- (a) 让模型从用户评论里抽取"情感倾向 + 提到的商品 + 打分",结果存进数据库
- (b) 用户说"帮我查订单 A123",要真的去查数据库
- (c) 用户问"你们几点下班"

---

## 参考答案(做完再看)

**A1** 因为 **LLM API 是无状态的**——模型不记得上一轮,所以每次请求都必须把**全部历史重发一遍**。
第 N 轮发送 N 轮的量,累计 = 1+2+…+N ≈ N²/2,故为平方级。

**A2** 至少两点:
①**贵**:平方级增长依然存在,窗口大只是"装得下",不是"不花钱";②**慢**:输入越长首 token 延迟越高;
③**⚠️质量下降**:长上下文存在 **"lost in the middle"** 现象——模型对开头和结尾利用得好,
**中间信息容易被忽略**。塞满 128K 未必比塞 10 轮精华答得好,可能更差。
> 核心:**窗口大 ≠ 该塞满。**

**A3** OpenAI 兼容 API 的响应里,`content` 和 `tool_calls` 是**并列的固定字段**,没有位置放第三个
自定义字段。想塞只能:让 content 输出 JSON(会与 tool_calls 打架,且用户可见回复要从 JSON 抠出来),
或另做一个 summarize 工具(那就变回工具调用了)。

**A4** 问题:①"尽量"是模糊限定词,给模型留了偷懒空间 ②"帮助用户"太泛,没说清什么场景用什么工具
③"不要出错"是无效指令——没有可执行含义,模型不知道要避免什么具体行为 ④没有输出格式约定。
改写示例:
> "你是电商客服助手。**任务**:回答订单、退款、账户相关问题。
> **必须**:①涉及订单状态时,必须先调用 query_order 查询,不得凭记忆或猜测回答;
> ②用户透露姓名/快递偏好等信息时,必须调用 remember 保存后再回复。
> **禁止**:①禁止编造订单信息;②禁止只在回复里声称'已查询/已记录'而未实际调用工具。
> **输出**:简洁中文,不超过 3 句话。"
用到的原则:去模糊限定词、给具体场景、规定顺序、点名禁掉已知失败模式、指令分区加权、约定输出格式。

**A5**
- ① prompt 请求:**无保证**,模型可能加代码块标记、解释、格式跑偏。
- ② JSON mode:保证输出**是合法 JSON**,但**不保证字段名/类型/必填项正确**。
- ③ schema 强制:保证**严格符合给定 JSON Schema**(字段、类型、必填都被保证)。
- **Function Calling 就是结构化输出**:你给模型 `parameters` 的 JSON Schema,模型返回严格符合
  该 schema 的 `arguments`——本质就是"按 schema 输出结构化 JSON",只是 API 把它包装成了
  "工具调用"这个更好用的概念。

**A6**
- (a) **结构化输出**——只是数据,要被代码消费(存数据库),不触发任何动作。
- (b) **function calling**——要真的触发查询动作。
- (c) **普通文本**——直接给人看,别过度设计。
