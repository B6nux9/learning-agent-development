# 上半场:上下文管理

## 0. 还 L2 的债
学习者 L2 自己追问出三件事:"LLM 无持续连接、追问会忘、**messages 越滚越长**"。
前两件 L5 已收(短期/长期记忆),第三件本节正面处理。

## 1. 为什么必然会爆:平方级的账
API 无状态 → **每轮都要重发全部历史**。后果:
```
第 N 轮请求发送 N 轮的量  →  累计 = 1+2+…+N ≈ N²/2
```
**token 消耗是平方级增长,不是线性。** 聊 10 轮的成本约是聊 5 轮的 4 倍。

拿他的 cli_agent.py 粗估:工具 schema 5 个≈600 token(每轮重发)+ system≈200(每轮重发)
+ 每轮往返≈200(累积)→ 第 50 轮单次请求 ≈ 10800 token,整个会话累计十几万。

## 2. 四个后果(第 4 个最反直觉)
1. **超限**:超出窗口 → 报错或被无声截断
2. **变贵**:平方级
3. **变慢**:输入越长首 token 延迟越高
4. **⚠️ 质量下降**:业界称 **"lost in the middle"** —— 上下文很长时,模型对**开头和结尾**
   利用得好,**中间的信息容易被忽略**。塞 50 轮未必胜过塞 10 轮精华,**可能更差**。

> **核心认知:窗口大 ≠ 该塞满。** 换个 128K 窗口只解决"装不装得下",
> 不解决"贵、慢、找不到重点"。

## 3. 心智转变:上下文是一份"预算表"
别把 messages 当"对话记录",当成**每轮要精心编排的有限预算**:

| 占位 | 来源 | 特性 |
|---|---|---|
| system prompt | 人设 + 注入的长期记忆(L5) | 固定,每轮重发 |
| 工具 schema | TOOLS(L3/L4) | 固定,每轮重发,**工具越多越胖** |
| 对话历史 | 累积 messages | **只增不减,罪魁祸首** |
| 工具返回结果 | `role: tool` 内容 | 常被忽略,**可能很大**(一大坨 JSON) |
| RAG 检索结果 | 知识库召回(L7) | 下节加入,也不小 |
| 当前用户输入 | 用户这句 | 小 |

**这就是 Context Engineering 的全部内容**(沃孚 JD 把 Prompt/Context/Harness Engineering 并列):
> 在有限预算里,决定每轮放什么、放多少、以什么形式放。

## 4. 四类管理策略
| 策略 | 做法 | 代价 |
|---|---|---|
| **① 截断** | 只留最近 N 轮 | 零成本;**早期信息永久丢失** |
| **② 摘要** | 老对话压缩成一段摘要替换原文 | 保主干;**额外 LLM 调用**(钱+延迟),有损 |
| **③ 混合(最常用)** | 最近 N 轮原文 + 更早的摘要 | 兼顾细节与长程;实现稍复杂 |
| **④ 外置** | 不放上下文,需要时检索回来 | 最省;**= L5 长期记忆 + L7 RAG** |

注意 ④ 与已有知识对上:**长期记忆存 memory.json 而不堆在 messages 里,本质就是"外置"**。

**常被忽略的便宜优化**:压缩工具返回结果——只回传模型需要的字段,别把一大坨 JSON
整个塞进 messages 长期占位。

---

## 5. 摘要的触发时机(学习者提问引出的讨论)
学习者提议:**每轮**让模型顺带输出一段摘要(像 tool_calls 一样不给用户看)。
- **对的一半**:复用已有调用、不另开 LLM 调用 —— 是 L5"把判断折进 tool use"的正确迁移。
- **错的一半**:①绝大多数轮次没爆、白付 output token(通常比 input 贵) ②摘要自身也会累积,
  每轮替换原文则反复有损压缩、越压越失真 ③任务混淆(回复+调工具+写摘要),稀释主任务质量
  ④**技术上不好实现**:OpenAI 兼容响应里 `content` 与 `tool_calls` 是并列固定字段,加不了第三个;
  想塞只能让 content 输出 JSON(与 tool_calls 打架)或另做 summarize 工具(又变回工具调用)。
  → **这个"想一次输出多个结构化字段"的诉求,正是下半场结构化输出要解决的问题。**
- **工业常用解:阈值触发**。平时不管,当 messages 的 token 数超过阈值(如窗口 60~70%)或轮数超限
  才压一次,把最老 N 轮压成摘要替换原文,保留最近几轮原文 = 策略③混合。**惰性优化思路。**

# 下半场:Prompt Engineering + 结构化输出

## 6. Prompt Engineering:从学习者 L5 的翻车提炼
L5 的 remember 不被调用,prompt 从"必要时...记住用户偏好"改成强命令后解决。对比两版可提炼全部要领:

| 改动 | 原则 |
|---|---|
| "必要时"→"**必须**" | **别用模糊限定词**("可以/尽量/必要时"= 给模型留偷懒空间) |
| 泛指→"姓名、快递偏好等" | **给具体例子**,别让模型猜边界 |
| 加"先保存,再回复" | **规定顺序**,多步任务明确先后 |
| 加"绝对不要只嘴上说…" | **点名禁掉已知失败模式**(负面约束) |
| 加"重要规则:" | **给指令分区/加权**,让关键约束跳出来 |

> 最值钱的一条:**发现模型有反复出现的错误行为,最有效的办法是在 prompt 里把该行为点名禁掉。**
> 泛泛说"要认真"没用,说"不要只说不做"才有用。

### Prompt 五块骨架
1 角色 · 2 任务 · 3 **约束(必须做/绝不能做 —— 大多数人写太少,这里最值钱)** · 4 示例 · 5 输出格式

### 三条实用原则
1. **prompt 是迭代出来的,不是一次写对的**(L4 调 router、L5 调 remember 都是改了才好)。面试可讲迭代过程。
2. **改 prompt 前先看模型实际干了什么**(tool_calls / 原始返回)。盲改是最常见的浪费。
3. **prompt 每轮重发,也是上下文预算的一部分** → **简洁而精确 > 长而模糊**(与上半场的账连上)。

## 7. 结构化输出 / 输出控制(补 L1-L5 缺口 2)
> JD 依据:蔚蓝"理解 prompt 设计与**输出控制**的基本逻辑";拼多多"**输出过滤**"。

**为什么 agent 特别需要**:聊天机器人的输出给人看,格式无所谓;agent 里模型输出常要**被代码消费**
(取值、判断、路由),格式不稳定就是灾难。

**学习者已手工做过**(L4 `route_llm`):①prompt 里求它只输出类别名 ②`.strip().lower()` 洗它
③`label in VALID_GROUPS else "order"` 验它+兜底。**求它/洗它/验它** = 没有工具支持时的土法,能用但脆。

### 三个层次(弱→强)
| 层次 | 做法 | 保证强度 |
|---|---|---|
| ① Prompt 请求 | prompt 里说"请输出 JSON" | ❌ 无保证(可能带 ```json、加解释、跑偏) |
| ② JSON mode | API 参数要求必须合法 JSON | ✅ 保证是合法 JSON,❌ 不保证字段对 |
| ③ 结构化输出/Schema 强制 | 提供 JSON Schema,API 保证严格符合 | ✅✅ 字段名/类型/必填项都被保证 |

③ 是当前主流(各家叫法不同:structured outputs / response_format / json_schema),**能用就用**,
省掉自写解析与校验的胶水代码。

### 串联知识的洞察
> **Function Calling 本质就是结构化输出的一个应用。**
> L3 学的:给模型 `parameters` 的 JSON Schema,模型返回严格符合的 `arguments` ——
> 这就是"按 schema 输出结构化 JSON",只是 API 包装成了"工具调用"这个更好用的概念。
> **所以学习者早就在用结构化输出了,只是没意识到。** 这也解释了为什么"想让模型多输出一个
> summary 字段"会和 tool_calls 打架 —— **它俩在抢同一套机制**。

### 什么时候用哪个(面试会问)
- 输出**要触发动作** → function calling
- 输出**只是数据、要被代码消费** → 结构化输出(抽取实体、打分、带理由的分类)
- 输出**直接给人看** → 普通文本,别过度设计
