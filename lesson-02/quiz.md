# L2 Quiz:手写 agent loop

> 出 quiz 时即落成此文件。下方含题目、学习者作答、教练反馈。结果:**5/5 达标**。

---

**Q1(概念)** agent loop 和 L1「一次 API 调用」最本质的区别?

- **学习者答**:agent loop 在一个 while 之中,LLM 不喊停或外部条件不叫停就不会停。
- **反馈**:✅ 对。核心是「循环 vs 一次」——不满足停止条件就反复决策。

**Q2(判断)** 「模型调用 query_order,是模型自己连上数据库去查的」——对吗?

- **学习者答**:不对。模型不会连数据库;LLM 只是有「可调用工具」,按需点单后,由 agent 在本地用该工具执行。
- **反馈**:✅ 完全正确,措辞已很专业(LLM 决策 / agent 本地执行)。面试标准答案。

**Q3(找 bug)** 删掉 `messages.append(msg)`(模型的点单消息),只保留 append tool 结果,会怎样?

- **学习者答**:tool 消息里的 tool_call_id 和前面上下文对不上,LLM 会报错。
- **反馈**:✅ 对。精确说:tool 结果带的 tool_call_id 没有任何前置 assistant 消息声明过 → API 报「tool 消息必须回应一条带 tool_calls 的前置消息」。

**Q4(停止条件)** loop 靠什么跳出?若模型每轮都点单、永不给答案会怎样?怎么兜底?

- **学习者答**:(不确定,认为代码里没有终结条件)理解为①用户自行结束 ②LLM 判断结束。并追问:LLM 判断结束后连接会不会被切断、上下文会不会丢?若用户没理解再追问,LLM 会不会忘了之前?若一直点单会烧 token,应加循环次数上限。
- **反馈**:
  - **纠正**:代码里**有**终结条件——`if not msg.tool_calls: return msg.content`(第 85–86 行)。`while True` 靠 `return` 结束,停止条件是「模型不再点单」。
  - **循环次数上限**:✅ 正确的生产兜底。
  - **两个追问含金量很高**:
    - LLM **没有持续连接**,每次 create 都是无状态一次性请求;「记得」是因为每轮重发整个 messages。
    - 当前代码里 messages 在 `run_agent` 内部每次从头 new,函数 return 就丢掉 → 再调一次会**完全忘光**上一轮。这正是 **L5 记忆与状态**要解决的。学习者靠直觉提前推到了这里。

**Q5(串联未来)** messages 越 append 越长会怎样?引出哪节课的问题?

- **学习者答**:数组越来越长,上下文变长会导致上下文丢失、回复不准确?
- **反馈**:✅ 方向对。精确为三个问题:①撞上下文窗口上限(截断/报错);②每轮重发全部历史 → token 成本+延迟线性上涨;③内容太长「lost in the middle」注意力变弱、回复变差。→ 对应 **L6 上下文长度管理**。

---

## 总评
达标 ✅,且超出 Beginner 深度:不仅写通 loop,还自行推导出 L5(记忆)、L6(上下文)两个后续核心问题。
