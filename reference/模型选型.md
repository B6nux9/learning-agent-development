## 一、最重要的一条分野:推理模型 vs 普通模型

这是你在 **L4 亲身撞过的坑**——`deepseek-v4-flash` 是推理型,`max_tokens=5` 被思维链吃光,
`content` 返回空串,导致 LLM 路由静默失灵。当时是当 bug 修的,这里把它变成系统认知。

| | **普通模型**(chat/instruct) | **推理模型**(reasoning / thinking) |
|---|---|---|
| 行为 | 直接生成答案 | **先在内部生成一长串推理过程,再给答案** |
| 延迟 | 低 | **高**(思考也要时间) |
| 成本 | 低 | **高**(推理过程也算 output token,且往往很长) |
| 强项 | 对话、分类、抽取、改写、常规问答 | 数学、复杂逻辑、多步规划、疑难代码 |
| 坑 | —— | ①`max_tokens` 设小会把答案挤没 ②延迟不可控 ③流式体验差 |

**选型判断（一句话）：要"想得深"用推理模型，要"反应快"用普通模型。**

### 落到 agent 里的实战规则

1. **路由/分类/意图识别 → 一律用普通模型的小/快档。**
   你只要一个词的输出,推理模型纯属浪费,还每轮加一次不可控延迟。(= 你 L4 最后的修法)
2. **主对话/工具调用 → 普通模型通常够用。** 工具调用本身是结构化任务,不需要深度思考。
3. **复杂任务规划(把大任务拆成多步) → 才考虑推理模型。** 这是 L8 Planning 会碰的。
4. **同一个 agent 里可以混用不同模型**——这本身就是**成本/延迟优化的加分回答**。
   (你 L4 的 capstone 就是这么干的:主对话一个模型、router 另一个。)

### 一个必须记住的坑
**推理模型的 `max_tokens` 要给足**——思维链会先消耗预算。设小了不是"答案被截断",
而是**答案根本没机会生成**,你拿到空字符串。(这就是你 L4 那个 `原始=''`。)

## 二、模型选型的 6 个维度(面试答题框架)

被问"你怎么选模型",按这 6 条讲,比背型号强得多:

1. **能力匹配**:任务要不要深度推理?要不要长上下文?要不要多模态?
2. **成本**:input / output token 单价(**推理模型的思维链按 output 计费,很贵**);
   有没有 **prompt caching**(重复的 system prompt / 工具 schema 能缓存,agent 场景省很多)。
3. **延迟**:首 token 延迟(TTFT)与总时长。用户直面的对话对延迟极敏感;离线批处理不敏感。
4. **上下文窗口**:决定你能塞多少历史/知识。**但窗口大 ≠ 该塞满**(见 L6:塞满既贵又降质)。
5. **工具调用能力**:能否稳定输出合法的 tool_calls、支持并行调用、支持结构化输出。
   **agent 场景这条权重极高**——模型再聪明,工具调用不稳定就没法用。
6. **部署与合规**:能否私有化/本地部署?数据出不出境?国内业务这条常常是硬约束。

## 三、主流厂商特性(粗粒度、按"性格"记,别背参数)

| 厂商 / 系列 | 大致定位与性格 | agent 场景注意 |
|---|---|---|
| **OpenAI (GPT 系列)** | 生态最全,SDK 与 function calling 是**事实标准**,文档成熟 | 大量第三方框架默认适配它;国内访问与合规是问题 |
| **Anthropic (Claude 系列)** | 长上下文与复杂指令遵循口碑好,**MCP 协议的提出方** | 学 MCP(L10)时会大量接触;国内同样有访问问题 |
| **阿里 (Qwen 通义千问)** | 国内生态最强之一,**开源 + 闭源双线**,尺寸齐全 | 国内业务与私有化部署的常见默认选项 |
| **DeepSeek** | 性价比突出,**OpenAI 兼容 API**(所以你的代码可直接迁移) | 你当前在用;推理型与普通型要分清(见第一节) |
| **Meta (LLaMA 系列)** | **开源权重**代表,可本地部署/微调的主力底座 | 涉及私有化、微调、vLLM 推理优化时的常客 |

> **重要认知:因为 OpenAI 的 API 形态成了事实标准,国内大多数厂商都提供"OpenAI 兼容"接口。
> 这意味着你在本课程写的所有代码,换个 `base_url` 和 `model` 名基本就能迁移。**
> 面试可以直接讲这一点——说明你理解的是接口抽象,不是绑死某一家。

## 四、面试怎么答"你怎么做模型选型"

一个好回答的结构(把你的真实经历套进去):

> "我会先分任务类型。比如我做的客服 agent 里,**主对话**和**意图路由**用的是不同模型:
> 路由只需要输出一个类别词,我就选快而便宜的普通模型;而且我踩过一个坑——
> 一开始路由用了推理型模型,`max_tokens` 设成 5,思维链把预算吃光,`content` 返回空串,
> 导致路由静默失灵、全部兜底到默认分组。定位之后换成非推理模型就好了。
> 这件事让我形成一个判断:**分类/路由这种要快要稳的活,不该用会'先想一会儿'的模型。**
> 更一般地,我会从能力匹配、成本(尤其推理模型的思维链按 output 计费)、延迟、
> 上下文窗口、工具调用稳定性、以及能否私有化部署这几个维度权衡。"

这个回答的杀伤力在于:**有真实踩坑、有归因、有形成的判断**,而不是背参数。
