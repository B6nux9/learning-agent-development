## 一、最有杀伤力的三条(优先背熟)

### 1. "你怎么保证 agent 真的执行了操作,而不是嘴上说说?"
**答:看 `tool_calls`,不看模型的话术。**

> 我做长期记忆时踩过一次:让模型在用户透露偏好时调 `remember` 工具落盘。跑完模型回我
> "我已经记下了您的姓名和快递偏好😊",看起来完全正常。但我去查持久化文件,**根本不存在**——
> `remember` 从头到尾没被调用过。模型只是**生成了一句像模像样的话,并没有执行动作**。
>
> 我是这样定位的:先查物证(文件在不在),再在工具执行处打印真实的 `tool_calls`,眼见为实。
> 根因是 **prompt 太软**——我原来只写"必要时可以记住用户偏好"。改成强命令、并**直接点名禁掉
> 偷懒话术**("绝对不要只嘴上说'已记住'却不调用工具"),模型就乖乖调了。
>
> 这件事给我一个判断:**agent 的验收必须基于可观测的动作,不能基于模型的自述。**
> 所以我做可观测性时,第一优先就是把每次工具调用完整记下来。

**加分延伸**:这直接引出为什么 agent 需要 tracing/日志,而不只是看最终回复。

---

### 2. "你怎么做成本 / 延迟优化?"
**答:分任务挑模型 + 只暴露该暴露的工具。**

> 两个我实际做过的:
>
> **(a) 分任务挑模型。** 我的客服 agent 里,主对话和意图路由用的是不同模型。路由只需要输出
> 一个类别词,我就用快而便宜的普通模型。**这里踩过一个坑**:一开始路由用了推理型模型,
> `max_tokens` 设成 5,结果思维链把 token 预算吃光,`content` 返回空字符串,路由静默失灵、
> 全部兜底到默认分组。换成非推理模型就好了。结论:**分类/路由这种要快要稳的活,
> 不该用会"先想一会儿"的模型。**
>
> **(b) 工具分组路由,不把全部工具塞给模型。** 工具一多有两个问题:①模型选择准确率下降
> ②**每一轮请求都要重复携带全部工具 schema**,既烧 token 又挤占上下文。我的做法是先用代码
> 判断用户意图落在哪个场景,本轮只暴露那一组工具。

### 3. "你用过 LangChain 之类的框架吗?"(**学完 L9 后再用这条**)
**答:用过,而且我是先手写过 agent loop 才去用框架的。**

> 我是从零手写 agent loop 起步的:自己实现 while 循环、解析 `tool_calls`、本地执行工具、
> 把结果以 `role: tool` 回填 messages、再喂回模型。所以我很清楚框架里
> AgentExecutor 那一层到底替我做了什么、哪些是它的抽象、哪些是模型本身的能力。
>
> 这让我在用框架时不至于"只会调 API"——比如出问题时我知道该去看哪一层。

## 二、核心概念的"我怎么理解"(用自己的话,别背定义)

### Agent 的能力从哪来?
**"能力来自工具,不来自 loop;可靠性来自约束。"**
- loop 本身很薄——就是"问模型 → 它要调工具就执行 → 结果回填 → 再问"。
- **链式调用(多步任务)是 loop 涌现出来的,不需要额外代码**:模型第一轮查订单,
  第二轮看到结果自己决定要不要退款。我当时零新增代码就支持了任意长的工具链。
- 但工具一多就得加约束(分组路由),否则选择准确率和 token 成本都崩。

### LLM 无状态,agent 怎么"记住"?
**没有魔法,就是"存下来 + 下次读出来拼进 prompt"。**
- **短期记忆**:一次会话内累积的 messages 列表。进程一死就没。
- **长期记忆**:自己实现"存 + 取"闭环——识别到值得记的事实写进持久层(文件/DB/向量库),
  新会话启动时读出来**注入 system prompt**。
- **注入要放 `system` 角色**,不能伪装成 user 消息:模型被训练成把 system 当作贯穿整段对话的
  背景设定;伪装成 user 容易被理解成"用户刚刚说了这句",多轮里串味。

### 记忆 vs 状态(容易被追问的区分)
**记忆是给模型看的,状态是给代码用的。**
- 我的防重复退款用一个 `REFUNDED` 集合,靠 `if order_id in REFUNDED` 判断——
  这份数据**从没进过 messages**,所以它是状态不是记忆。

### 错误处理:"错误也是信息"
工具抛异常不该让 agent 崩。用 try/except 接住,把错误文本作为 tool 结果**回传给模型**,
模型会自己调整策略:换参数重试、改调别的工具、或体面地告诉用户。
**你只需保证错误能以文字到达它眼前,自愈是白送的。**

## 三、踩坑清单(被问"遇到过什么难搞的 bug"时用)

### 1. 静默兜底掩盖 bug(最值得讲)
LLM 路由写了 `return label if label in VALID_GROUPS else "order"` 这样的校验兜底。
结果所有输入都被归到 `order`,但**程序不报错**——因为兜底把异常吞了。
定位方法:**在兜底之前把真实值用 `repr`/`!r` 打出来**,立刻看到 `原始=''`。

> **教训:fallback 是双刃剑——让程序不崩,也把 bug 藏起来。写兜底的地方,
> 调试期一定要能看见"兜底之前的真实值"。**

### 2. 漏写兜底 return → 静默返回 None
函数只在 `if` 分支里 return,忘了另一支,Python **不报错不警告,默默返回 `None`**,
然后在很远的地方以奇怪方式炸掉(我这次炸在 API 报 "content should be a string")。
> **自查动作:写完带返回值的函数,回头扫一眼"每个分支都 return 了吗"。**

### 3. messages 混血列表导致 AttributeError
`messages` 里同时躺着两种东西:自己手写的 system/user/tool 是 **dict**(`m["role"]`),
API 返回后 append 的 assistant 是 **SDK 对象**(`m.role`)。统一用一种方式访问必炸。
> 看到 `'dict' object has no attribute X` 或 `'X' object is not subscriptable`,
> 先想"我这个列表是不是混血了"。

### 4. 环境变量:VSCode 的环境快照坑
Windows 上用 `[Environment]::SetEnvironmentVariable(...,"User")` 设的永久变量,
**VSCode 读不到**——因为它在启动那一刻就把环境变量拍了快照,开新终端标签也没用,
**必须完全重启 VSCode**。

### 5. `ensure_ascii=False`
中文 JSON 不加这个参数会被转义成 `\uXXXX`。除了 debug 时看不懂,更实际的问题是:
**这些 JSON 要作为工具结果回传给模型,转义后平白多烧 token**(一个中文字可能从 1-2 token
变成 4-5 token)。

## 四、能讲的项目(简历素材)

### 命令行多工具客服 Agent(阶段一 capstone)
- 手写 agent loop,不依赖框架
- 5 个业务工具(查订单/退款/查退款进度/改地址/查积分),按场景**分组路由**
- **共享底座 + 场景增量**的分组策略:通用查询工具在每组都带上,专用工具按意图追加
- 会话内多轮记忆(能理解"把**它**退了"里的指代)
- 错误自愈:工具异常转成错误 JSON 回传,模型自行调整
- 循环次数上限兜底,防止模型抽风空转
- 路由做了两版对比:**关键词匹配**(零成本但硬编码) vs **LLM 分类**(灵活但每轮多一次调用),
  能讲清两者的取舍

### 跨会话长期记忆
- `remember` 工具 + JSON 持久层 + system prompt 注入,实现跨进程重启的用户记忆
- 设计取舍:**把"要不要记"折进 tool use 让模型自己判断**,而不是另开一次分类调用
  (省一次 LLM 调用)
