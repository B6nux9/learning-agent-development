# L4 Quiz:多工具编排

> 概念检验,答完我给反馈。达标后上 capstone。

**Q1(判断+解释)** 有人说:"工具多了就多写几个,反正 loop 不用改,模型自己会挑。"
这话哪里对、哪里不对?(提示:对在哪一层,崩在哪两个地方)

**Q2** 用你自己的话说清「模型编排」和「代码编排/路由」的区别。各举一个更适合用它的场景。

**Q3** "代码编排"里,开发者的代码具体多做了哪一步?它把什么东西"减负"给了模型?

**Q4(连回旧课)** 30 个工具全塞给模型,每轮都要带上全部 schema。这会直接恶化你之前自己
推导出来的哪个后续问题?(L2 你就想到过)

**Q5(设计题)** 一个客服 agent 有这些工具:查订单、查物流、发起退款、查退款进度、改收货地址、
查积分、积分兑换、修改密码、注销账户。如果让你做「静态分组」,你会分成几组、怎么分?
说清你的分组依据。

---

## 参考答案(做完再看)

**A1** 对:loop 确实不用改,能力来自工具(L3 抽象)。不对/崩点:①选择困难——工具越多
description 路由质量越差,模型选错漏选;②上下文膨胀——每轮都塞全部 schema,烧 token 又挤占
对话历史。所以"多写几个"到二三十个就不成立了。

**A2** 模型编排:工具全给,模型自己在全集里决定调哪个、什么顺序(L2/L3)。代码编排:开发者代码
先按意图筛,本轮只把相关子集给模型。模型编排适合工具少/探索性 demo;代码编排适合工具多/生产/
要可靠可控。

**A3** 多做了"意图 → 工具子集"的判断这一步(路由)。把"在几十个工具里选对一个"这个负担减掉了,
模型每轮只在 2-3 个相关工具里选,又准又省 token。

**A4** 上下文长度管理(L6)——也和成本/延迟相关。你在 L2 就自己推导出"messages 越滚越长"。

**A5** 无标准答案,看分组依据是否成立。一种合理分法:
- 订单/物流组:查订单、查物流
- 售后组:发起退款、查退款进度、改收货地址
- 账户组:查积分、积分兑换、修改密码、注销账户
依据:按"用户来干什么"(场景/意图)聚类,同组工具常一起被用到、且暴露给模型时彼此不易混淆。
(注:修改密码/注销这类高危操作,后面 L10 会讲还要加确认/权限约束,这里先只做分组。)
