# L4 讲义:多工具编排(阶段一收官课 + capstone)

> L2/L3 学会了"放养":工具全丢给模型,loop 自己涌现链式调用。L4 回答:
> 工具从 2 个变 20、30 个,放养不够了怎么办。从「能跑」到「能上生产」的分水岭。

## 0. 本节目标
- 讲清工具变多时崩在哪(还 L3 埋的债:①选择困难 ②schema 每轮吃 token)
- 新抽象:**编排的「决策权」放哪**——模型编排 vs 代码编排/路由
- 进阶谱系:全放养 → 静态分组 → 动态检索(tool RAG,埋 L7 伏笔)
- capstone:把 L2~L4 综合成一个能交互的命令行多工具客服 agent

## 1. 工具变多,崩在两个地方(还 L3 的债)
- **① 选择困难**:工具越多,description 路由质量越差,模型选错/漏选/在相似工具间横跳。
  经验阈值:单次暴露给模型的工具 > ~10 个,选择准确率明显下降。
- **② 上下文膨胀**:每轮请求都要塞【全部】TOOLS schema。30 个工具几千 token,每轮重复烧,
  还挤占对话历史空间。(这条直连 L6 上下文管理。学习者 L3 Q5 只答出①,②是补的。)

## 2. 最重要的新抽象:编排的「决策权」放哪?
L3 说"能力来自工具,不来自 loop"。L4 配一句:**"可靠性来自约束"**。

- **模型编排(model-driven)**:工具全给,模型自己在全集里挑。= L2/L3 做法。灵活但工具一多就崩。
- **代码编排 / 路由(code-driven)**:你的代码先按意图筛,本轮只把【相关子集】丢给模型。
  模型每轮只在 2-3 个工具里选,又准又省 token。模型依旧在子集内自由链式编排(L3 涌现照旧)。

代码编排 ≠ 抢模型的活,是给它减负:先判断意图落哪个场景,本轮只暴露那个场景的工具。

## 3. 进阶谱系(点到即止,埋 L7 伏笔)
1. **全放养**:所有工具全给。demo 够用。
2. **静态分组**:工具按场景/角色分组(查询组/售后组/账户组),先判意图落哪组,只给那组。← L4 主战场
3. **动态检索(tool RAG)**:工具几百个时,工具描述做成向量库,按 query 检索 top-k 工具再给模型。
   = 学习者 L3 结尾自己想到的"按需检索工具",思路同 L7 RAG。到 L7 回来点。

## 4. capstone:命令行多工具客服 agent
把 L2(loop)+ L3(多工具/错误自愈/循环上限)+ L4(分组路由)综合:
一个能在终端里【多轮对话】的客服 agent,支持查订单 + 退款 + 至少一个新场景(如查账户/改地址),
用【分组或路由】管理工具,而不是一股脑全给。详见 homework 骨架。
